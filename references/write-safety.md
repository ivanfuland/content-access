# 写操作安全规范

所有平台写操作（发帖、回复、点赞、删除、签到等）在执行前必须经过二次确认。

---

## 二次确认流程

1. **识别写操作**：用户请求包含 post / reply / like / delete / sign-in / daily 等动词，或明确说"发推"、"回复"、"点赞"、"删帖"等
2. **展示确认信息**：向用户清晰展示即将执行的操作内容
3. **等待确认**：收到明确的"确认"/"yes"/"是"/"执行"后才运行命令
4. **拒绝处理**：用户取消则终止，不执行任何命令

---

## 确认信息模板

### 发帖（twitter post）

```
⚠️ 即将发布推文：

内容：「<tweet_text>」
平台：X (Twitter)
账号：当前登录账号

确认发布吗？（yes / 取消）
```

### 回复（twitter reply）

```
⚠️ 即将回复推文：

回复目标：<tweet_url>
回复内容：「<reply_text>」
平台：X (Twitter)

确认回复吗？（yes / 取消）
```

### 点赞（twitter like）

```
⚠️ 即将点赞推文：

目标：<tweet_url>
平台：X (Twitter)

确认点赞吗？（yes / 取消）
```

### 删除（twitter delete）

```
⛔ 即将删除推文（不可恢复）：

目标：<tweet_url>
平台：X (Twitter)

确认删除吗？（yes / 取消）
```

### V2EX 签到（v2ex daily）

```
ℹ️ 即将执行 V2EX 每日签到领取铜币。

确认执行吗？（yes / 取消）
```

### 通用写操作模板（其他平台）

适用于 Twitter 以外的所有平台写操作（小红书发布、Instagram 点赞/评论、即刻发帖、BOSS 打招呼、Reddit 评论等）。

```
⚠️ 即将执行写操作：

平台：<platform>
操作：<command>（如 xiaohongshu publish / instagram like / jike create）
内容：<完整内容或操作目标，不可省略>

确认执行吗？（yes / 取消）
```

删除类操作使用 ⛔ 并强调不可恢复：

```
⛔ 即将删除内容（不可恢复）：

平台：<platform>
目标：<内容或 URL>

确认删除吗？（yes / 取消）
```

## 风险分级

| 风险级别 | 操作类型 | 图标 | 说明 |
|---------|---------|------|------|
| 高风险 | 删除操作 | ⛔ | 不可恢复，需明确确认 |
| 中风险 | 发帖/回复 | ⚠️ | 公开可见，发送前展示完整内容 |
| 低风险 | 点赞/收藏/签到 | ℹ️ | 可撤销，简要确认即可 |

---

## 注意事项

- 确认信息必须包含**完整的操作内容**，不能只说"即将发帖"而不展示内容
- 批量写操作（如批量点赞）需额外强调操作数量
- 若用户在一次对话中已确认同类操作，**不得复用上次确认**，每次操作独立确认
- 确认词匹配：`yes`、`是`、`确认`、`执行`、`ok`、`好`；拒绝词：`no`、`取消`、`不`、`算了`
